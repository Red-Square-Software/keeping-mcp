# Phase 2.5 Plan 01: Timer Status Read Tool Summary

**One-liner:** Shipped `keeping_timer_status` MCP read tool backed by `GET /{orgId}/time-entries/last` with strict single-key wrapper extractor and a 404-as-graceful-empty branch, closing the TIMER-01 status-read portion in a single RED → GREEN → WIRE vertical slice.

## Outcome

A MCP client calling `keeping_timer_status` receives `{ time_entry: <raw inner entry>, is_running: <boolean> }` where `is_running` is derived from `entry?.ongoing === true`. New users with zero historical entries (404) and 200-OK responses with no usable `time_entry` both collapse to `{ time_entry: null, is_running: false }` with no `isError` key — graceful empty, not error. 401, MultiOrg, 403/5xx, and network errors flow through the existing `toIsErrorContent` envelope unchanged. The Phase 2 `KeepingClient` is reused verbatim — no new method, no new error class, no client surface drift.

End-to-end verification: a fresh `npx keeping-mcp` (after `npm run build`) now exposes six read tools — the five Phase 2 reads (`keeping_me`, `keeping_organisations`, `keeping_projects`, `keeping_tasks`, `keeping_list_entries`) plus `keeping_timer_status`.

## Patterns Established for Phase 3

The Phase 2.5 work introduces three patterns Phase 3 timer writes can reuse:

1. **Strict single-key wrapper extractor** — `extractTimeEntry(raw): Record<string, unknown> | null` is a reusable template for Phase 3's `keeping_resume_timer` which will need to read the most-recent stopped entry's `id` before posting `/time-entries/{id}/resume`. The strict-vs-tolerant trade-off is locked: when the OpenAPI spec is authoritative (post-Phase-2 probe), prefer strict + schema-drift test over tolerant fallback.
2. **404-as-graceful-empty branch** — the `if (err instanceof KeepingApiError && err.status === 404) { return <static empty payload>; }` shape is the template for any "no entry yet → still a valid answer" tool. `keeping_resume_timer` may use the same shape if "no recent entry to resume" is a valid sentinel result rather than an error.
3. **TDD vertical slice** — RED test file with module-not-found gate → GREEN minimal impl → WIRE into composition root. Three commits per vertical slice keeps each step independently revertable and pairs each test cohort with its impl in git history.

## Notes for Phase 3

- The `is_running` boolean Phase 2.5 derives is exactly the input Phase 3's `keeping_stop_timer` / `keeping_resume_timer` chain logic will consume. Phase 3 can call `keeping_timer_status` first to decide which write to dispatch.
- The OpenAPI-locked singular `time_entry` wrapper shape is now exercised by 10 unit tests. Schema drift will surface as a Test 1/2 failure (entry not preserved verbatim) or Test 9 failure (path drift).
- The `KeepingApiError.status` field continues to be the canonical discriminator for graceful-vs-error branches across tools (Phase 2's projects/tasks "feature not enabled" + Phase 2.5's "no entry yet" are the same pattern).
