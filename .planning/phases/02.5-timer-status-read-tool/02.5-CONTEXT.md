# Phase 2.5: Timer Status Read Tool - Context

**Gathered:** 2026-06-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Ship a single read-only MCP tool: `keeping_timer_status`. The tool calls
`GET /{org_id}/time-entries/last` with no query parameters, returns the raw
`time_entry` object the Keeping v1 API delivers, and adds a derived top-level
`is_running` boolean from `time_entry.ongoing`. No writes. No new request-path
strategy. No new client capabilities. Phase 2's `KeepingClient`, identity
cache, `resolveOrgId()`, tool annotation pattern, and `toIsErrorContent()`
error envelope are reused verbatim.

Requirements covered: **TIMER-01** (status-read portion only — `start`/`stop`/`resume`
write tools remain in Phase 3 per D-33-R).

**Out of scope (Phase 3, not 2.5):**
- `keeping_start_timer`, `keeping_stop_timer`, `keeping_resume_timer` (writes).
- Any filter-param surface on `/time-entries/last` beyond pure status — see D-2.5-05.
- `user_id` input for admin/team-scope multi-user queries — see D-2.5-06.

</domain>

<decisions>
## Implementation Decisions

### Response Shape

- **D-2.5-01:** Wire shape is `{ time_entry: <raw API entry>, is_running: boolean }`.
  The API's `time_entry` wrapper key is preserved literally (D-34-R pass-through
  spirit). `is_running` is added as a TOP-LEVEL sibling — not merged into the
  entry. SC #1 reads "raw `time_entry` object plus top-level `is_running`" —
  this shape is the literal match.
- **D-2.5-02:** `is_running` derivation is `time_entry?.ongoing === true`. Per
  SC #4: `ongoing: true` → `is_running: true`; `ongoing: false` → `is_running:
  false`; missing `ongoing` (or missing `time_entry`) → `is_running: false`.
  Defensive default — never throws on missing field.

### Empty / 404 Handling

- **D-2.5-03:** A 404 from the endpoint (documented in OpenAPI — "no matching
  entry") collapses to a graceful empty response: `{ time_entry: null,
  is_running: false }`. New users with zero historical entries get a usable
  answer rather than `isError`. Mirrors Phase 2's "graceful empty for
  feature-disabled orgs" pattern (META-01, META-02).
- **D-2.5-04:** A 200 OK whose body fails to yield a usable `time_entry`
  (key missing, `null`, or non-object) collapses to the same graceful empty
  shape as D-2.5-03. One empty-state surface, regardless of cause.
- **D-2.5-04a:** 404 detection happens by inspecting the thrown
  `KeepingApiError.status === 404` inside the tool handler's `catch`. All
  other errors (401, 403, 5xx, network) keep flowing through
  `toIsErrorContent()` unchanged. The empty-state branch is the ONLY
  special-case catch in the tool.

### Wrapper Normaliser Strategy

- **D-2.5-05a:** Strict wrapper read — the tool reads `raw.time_entry`
  verbatim and nothing else. No defensive multi-key fallback (no `entries[0]`,
  no bare-entry-as-object acceptance). entries-list.ts needed a tolerant
  normaliser because its schema was unverified pre-D-34-R; here the OpenAPI
  spec is authoritative and the wrapper is locked. Drift will be caught by
  the schema-drift test rather than masked by tolerant parsing.

### Filter Params & Input Surface

- **D-2.5-05:** Pure-status semantics — the tool sends NO query parameters
  to `/time-entries/last`. Endpoint hit is bare `/{orgId}/time-entries/last`.
  Returns the absolute most-recent entry (running or stopped) for the
  authenticated user. The `is_running` boolean is what answers "is a timer
  running NOW?" — derived client-side from `ongoing`. Rejected alternative
  `?ongoing=true` because it would lose the most-recent-stopped-entry data
  Phase 3's `keeping_resume_timer` will need.
- **D-2.5-06:** Tool input schema is `{ organisation_id?: string }` — and
  nothing else. No `user_id`. No `date`. No `purpose`/`project_ids`/etc.
  The endpoint's other 8 documented filters are intentionally NOT exposed.
  Admin/team-scope multi-user queries are out of v1 scope (`KEEPING_TOKEN`
  is a personal access token in the dominant single-developer use case).

### Tool Wiring & Conventions

- **D-2.5-07:** File path `src/tools/timer-status.ts`, registration function
  `registerTimerStatus(server, client)`, registered by name `keeping_timer_status`.
  Pattern is byte-for-byte parallel to `src/tools/entries-list.ts`.
- **D-2.5-08:** Tool annotations are `readOnlyHint: true, destructiveHint:
  false, idempotentHint: true, openWorldHint: true` (matching every Phase 2
  read tool; the openWorldHint is required because we call an external API).
- **D-2.5-09:** Dispatch via `client.get<unknown>(path)`. No new
  `KeepingClient` method, no new request strategy. `resolveOrgId(input.organisation_id)`
  resolves multi-org per D-26..29.
- **D-2.5-10:** Multi-org error envelope (D-27) and mid-session 401 envelope
  (D-25) are inherited unchanged. `MultiOrgError` and `KeepingApiError`
  surface through `toIsErrorContent()` as today.

### Test Strategy

- **D-2.5-11:** Unit tests are the only verification surface. No live-API
  probe needed (the endpoint is already proven by the Phase 2 probe-live
  run; the OpenAPI mirror at `.planning/research/keeping-openapi.json`
  documents the contract).
- **D-2.5-12:** Test fixture reuses the existing
  `test/fixtures/time-entry-response.sample.json` SHAPE as the structural
  reference. Tests construct mock `client.get` responses by wrapping a single
  fixture entry in `{ time_entry: ... }`. SC #5 satisfied.
- **D-2.5-13:** Test cases (minimum):
  1. `ongoing: true` → `is_running: true`.
  2. `ongoing: false` → `is_running: false`.
  3. Missing `ongoing` field → `is_running: false`.
  4. 404 from client (mocked as `KeepingApiError` with `status: 404`) →
     `{ time_entry: null, is_running: false }`.
  5. 200 OK with `{ time_entry: null }` → graceful empty (same as #4).
  6. 200 OK missing `time_entry` key entirely → graceful empty (same as #4).
  7. 401/500 error → flows through `toIsErrorContent` → `isError: true`.
  8. Multi-org error (no org input, no env, multiple orgs) → flows through
     `toIsErrorContent` → `isError: true`.
  9. Path assertion: client mock is called with exactly
     `/${orgId}/time-entries/last` (no query string).
  10. Annotations assertion: tool registers with the four locked hints.

### Claude's Discretion

- Exact wording of the `description` (the AI-facing surface). Suggest:
  "Returns the most recent time entry for the authenticated user plus a
  derived `is_running` boolean indicating whether a Keeping timer is
  currently running. Read-only; no API mutation. Use this to decide whether
  a `keeping_stop_timer` or `keeping_resume_timer` call is appropriate
  (Phase 3)." — planner is free to refine.
- Whether to ship an `outputSchema` on this tool. Default: defer (consistent
  with Phase 2 read-tool deferral noted in `02-CONTEXT.md` Deferred Ideas).
  Planner may revisit if it discovers a low-friction path.
- Whether the registration is wired in `src/server.ts` next to the Phase 2
  read tools or in its own dedicated section — planner decides the file
  topology.
- Whether to add a fresh fixture file at
  `test/fixtures/time-entry-last.sample.json` mirroring the single-entry
  wrapper shape, or to construct the wrapper inline in tests. Either is
  acceptable; fixture file is slightly more discoverable, inline is
  slightly less file-thrash.

</decisions>

<specifics>
## Specific Ideas

- Tool name (locked): `keeping_timer_status` — matches ROADMAP, matches
  the deferred-idea entry in `02-CONTEXT.md`.
- Empty-state response (exact shape): `{ time_entry: null, is_running: false }`.
- Multi-org error message wording: inherits the D-27 exact format from
  Phase 2 — `Multiple organisations available. Pass organisation_id, or set
  KEEPING_ORG_ID. Options: <id> (<name>), ...`.
- Mid-session 401 wording: inherits D-25 exact format from Phase 2.
- Tool description should mention the Phase 3 sibling tools by name
  (`keeping_stop_timer`, `keeping_resume_timer`) — the AI client uses this
  tool to decide which write tool to call next.
- Endpoint URL form: bare `/{orgId}/time-entries/last` (no trailing slash,
  no query string).

</specifics>

## Deferred Ideas

  read-tool deferral. Revisit once the singular-wrapper shape is locked

  contract). Could be added in a future schema-drift CI cycle.
- Auto-trigger Phase 3 timer writes from a running `is_running: true` —
  product-level UX; belongs in Phase 3 + later UX iterations.
