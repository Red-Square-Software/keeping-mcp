## Response Shape

### Q1 — Top-level wire shape for keeping_timer_status response?

| Option | Description | Selected |
|--------|-------------|----------|
| Wrap + sibling | `{ time_entry: <raw API entry>, is_running: bool }`. Preserves API's exact wrapper key. SC #1 reads "raw time_entry object plus top-level is_running" — literal match. Easy mental model for AI clients. | ✓ |
| Flatten merged | `{ ...rawEntry, is_running: bool }`. is_running lives ON the entry. Loses API wrapper key; renames topology. Diverges from D-34-R pass-through spirit. | |
| Rename wrapper to 'entry' | `{ entry: <raw>, is_running: bool }`. Drops API's underscore key for terser name. Mild rename violates D-34-R top-level preservation. | |

**User's choice:** Wrap + sibling (Recommended).
**Notes:** Matches SC #1 literal reading. Preserves OpenAPI ground-truth wrapper key. Becomes D-2.5-01.

### Q2 — Defensive normaliser for the wrapper?

| Option | Description | Selected |
|--------|-------------|----------|
| Strict wrapper only | Read `raw.time_entry` verbatim. If missing/not-object, treat as empty-state. Tighter contract; surfaces drift loudly. OpenAPI spec is authoritative — only one documented shape. | ✓ |
| Tolerant like entries-list | Accept `{time_entry:{...}}` OR bare entry object OR `{entry:{...}}`. Defence-in-depth, but masks API drift. entries-list needed it because schema was still unverified pre-D-34-R. | |

**User's choice:** Strict wrapper only (Recommended).
**Notes:** OpenAPI is authoritative now; tolerant parsing would mask drift the schema-drift test should catch. Becomes D-2.5-05a.

## Empty / 404 Handling

### Q3 — 404 (no matching entry) surfaces as?

| Option | Description | Selected |
|--------|-------------|----------|
| Graceful empty | Catch 404 → return `{ time_entry: null, is_running: false }`. New users with zero entries get usable answer. Matches Phase 2 'graceful empty for feature-disabled orgs' pattern. | ✓ |
| isError envelope | Let `toIsErrorContent` surface 'no entry found' as isError. Consistent with other read tools that don't special-case 404. AI client must branch on isError vs success. | |
| Graceful empty + log | Same as Recommended but emit `log.info` via `server.sendLoggingMessage` so the AI client sees the empty-state signal on the protocol channel. | |

**User's choice:** Graceful empty (Recommended).
**Notes:** Becomes D-2.5-03. 404 detection via `KeepingApiError.status === 404` inside the tool's catch (D-2.5-04a).

### Q4 — 200 OK but shape doesn't yield a valid time_entry?

| Option | Description | Selected |
|--------|-------------|----------|
| Same as 404 — empty | Any path that fails to produce a usable entry collapses to `{ time_entry: null, is_running: false }`. One empty-state surface, regardless of whether 200+null or 404. Simpler contract. | ✓ |
| isError on shape mismatch | Only 404 maps to empty. Unexpected 200 shape → isError with 'unexpected response shape'. Surfaces API drift loudly but creates two empty-ish surfaces (404 vs shape-bug). | |

**User's choice:** Same as 404 — empty (Recommended).
**Notes:** Single empty-state surface; drift caught by schema-drift test, not by runtime branching. Becomes D-2.5-04.

## Filter Params Exposure

### Q5 — Semantic intent / filter strategy?

| Option | Description | Selected |
|--------|-------------|----------|
| Pure status, no filters | Input = `{ organisation_id? }` only. Always hits `/time-entries/last` with NO query params. Returns absolute most-recent entry (running or not). `is_running` derives from `ongoing`. Aligns with SC #1 + tool name semantics. Minimal API surface, easy to grow. | ✓ |
| Hard-coded ongoing=true | Server-side `?ongoing=true`. Tool name literally answers 'is a timer running NOW?' On 404 → `is_running:false`. BUT: loses 'most recent stopped entry' info needed for 'resume' UX in Phase 3. | |
| Expose ongoing as optional bool | Input `{ organisation_id?, ongoing?: bool }`. Caller chooses. Adds one knob. Forwards explicitly if set. Slight surface bloat but flexible. | |
| Full filter pass-through | Expose `date`/`purpose`/`project_ids`/etc. Treats `/time-entries/last` as general 'last-entry lookup'. Loses 'timer status' framing — becomes a different tool. Scope creep risk. | |

**User's choice:** Pure status, no filters (Recommended).
**Notes:** Preserves "most recent stopped entry" data Phase 3's `keeping_resume_timer` will need. Becomes D-2.5-05.

**Notes:** Becomes D-2.5-06. Multi-user querying deferred to a future team/admin tools phase.
