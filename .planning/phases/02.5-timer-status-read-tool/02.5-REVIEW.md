# Phase 02.5: Code Review Report

**Reviewed:** 2026-06-11
**Depth:** standard

## Summary

`keeping_timer_status` is a small, focused read tool that follows the
established Phase 2 register-pattern (matches `me.ts`/`entries-list.ts` for
schema, annotations, error envelope). The integration surface — `client.get`,
`resolveOrgId`, `KeepingApiError`, `toIsErrorContent` — is reused verbatim, so
nothing new touches auth, throttling, retry, or token-redaction code.

Functional behaviour matches the locked decisions (D-2.5-01..10) on the happy
paths covered by the 10 unit tests. No security defects: handler never logs,
the token is never read or referenced, all 401/4xx error envelopes route
through the existing sanitised `toIsErrorContent` path.

Two warnings worth fixing before merge:

- **WR-01** `extractTimeEntry` does not actually collapse a bare-array
  `time_entry` to `null` despite the source comment promising it does, which
  breaks the strict-wrapper-read contract (D-2.5-05a) and leaks an array under
  a key the response shape documents as a single object. Tests do not cover
  the array case so the gap is invisible.
- **WR-02** The source comment on `extractTimeEntry` references "the
  schema-drift test" as the loud-failure mechanism, but no such test exists
  and the function is not exported, so the documented escape hatch is fiction.

Three Info items cover documentation drift, a redundant-key test gap, and a
test-mock signature shortcut.

## Warnings

### WR-01: `extractTimeEntry` accepts arrays as `time_entry`, contradicting the documented strict-wrapper contract

**File:** `src/tools/timer-status.ts:58-63`

**Issue:** The source comment on `extractTimeEntry` (lines 17-22 and 53-56)
states that the strict wrapper "accepts ONLY when `raw.time_entry` is a non-null
object" and that "non-object, bare-array collapses to null". The implementation
does not honour the bare-array half of that promise:

```ts
function extractTimeEntry(raw: unknown): Record<string, unknown> | null {
  if (raw === null || typeof raw !== "object") return null;
  const candidate = (raw as Record<string, unknown>).time_entry;
  if (candidate === null || typeof candidate !== "object") return null;
  return candidate as Record<string, unknown>;
}
```

For `raw = { time_entry: [] }` or `{ time_entry: [{...}] }`:

- `typeof candidate === "object"` is `true` for arrays in JS.
- `candidate === null` is `false`.
- The guard passes; the array is cast to `Record<string, unknown>` and
  returned as if it were the inner entry.

Downstream, `entry?.ongoing === true` is `false` for an array, so
`is_running` is `false`, and the payload becomes
`{ time_entry: [...], is_running: false }` — an array surfaced under a key
the tool contract (D-2.5-01) documents as either a single object or `null`.
This is exactly the "drift masked silently" failure mode D-2.5-05a was
written to prevent (per Pitfall 12 cited in the source preamble).

Consumers downstream that branch on `payload.time_entry === null` for the
empty-state will misread an empty `time_entries` array (a plausible API drift
shape) as a present entry.

The test suite does not exercise any array shape for `time_entry`, so this
gap is invisible from the green-bar.

**Fix:** Add an explicit array check before the `typeof` guard:

```ts
function extractTimeEntry(raw: unknown): Record<string, unknown> | null {
  if (raw === null || typeof raw !== "object") return null;
  const candidate = (raw as Record<string, unknown>).time_entry;
  if (
    candidate === null ||
    typeof candidate !== "object" ||
    Array.isArray(candidate)
  ) {
    return null;
  }
  return candidate as Record<string, unknown>;
}
```

Add a unit test that asserts `{ time_entry: [] }` and `{ time_entry: [{...}] }`
both collapse to the graceful-empty `{ time_entry: null, is_running: false }`
response, mirroring Tests 5 and 6.

### WR-02: Source comment references a "schema-drift test" that does not exist; helper is not exported either

**File:** `src/tools/timer-status.ts:21-22, 56`

**Issue:** Two passages in the source promise a "schema-drift test" as the
loud-failure mechanism that justifies the strict (rather than tolerant)
wrapper read:

- Lines 21-22: "drift should fail loudly via the schema-drift test rather
  than be masked"
- Line 56: "drift should fail loudly via the schema-drift test, not be masked"

A repo-wide search for `schema-drift|schema_drift|schemaDrift` in `test/`
finds zero matches. The strict-wrapper-read function `extractTimeEntry` is
also not exported, so an external schema-drift test could not call it
directly — it would have to drive drift through the registered tool and the
MCP `InMemoryTransport` like the existing tests already do. The "loud
failure" promised by the comment is at best a generic re-derivation through
Tests 5/6 (which assert graceful empty, not loud failure).

This is documentation drift, not a runtime bug, but the comment is load-
bearing rationale for the design choice locked by D-2.5-05a — leaving it
unmoored makes the next maintainer reach for the tolerant fallback when the
real API shape inevitably differs in some edge case.

**Fix:** Either (a) add a real schema-drift test in
`test/tools/timer-status.test.ts` that asserts drift shapes produce a
distinguishable result (e.g., a wrapped error containing the unknown keys,
or an explicit drift signal in the payload), and reference that test name in
the comment; or (b) replace the "fail loudly via the schema-drift test"
phrasing with "fall through to graceful empty via Tests 5 and 6, which is
the loud failure surface for this tool" so the comment matches reality.

Option (b) is the cheaper fix and aligns the doc with what was actually
shipped.

### IN-02: Test 4 hard-codes a sanitised body that masks the constructor contract

**File:** `test/tools/timer-status.test.ts:104-106`

**Issue:** The 404 test passes a literal body to the `KeepingApiError`
constructor:

```ts
throw new KeepingApiError(404, '{"error":{"message":"No entry found"}}');
```

The constructor contract (per `src/keeping/errors.ts:1-7`) says
"`KeepingApiError` NEVER stores a raw API response body — callers must pass
it through `sanitiseBody(text, token)` first". The test happens to pass an
already-token-free string, so the sanitisation invariant is not exercised.
A future maintainer copy-pasting this test for a new error case might
inadvertently pass an unsanitised body and rely on the `***` redaction never
being hit.

**Fix:** Either (a) add a brief comment at the throw site noting the body
must already be sanitised, or (b) call `sanitiseBody(rawBody, "fake-token")`
inline so the test demonstrates the contract:

```ts
import { sanitiseBody } from "../../src/keeping/errors.js";
// ...
throw new KeepingApiError(
  404,
  sanitiseBody('{"error":{"message":"No entry found"}}', "fake-token"),
);
```

Low priority — pure documentation hygiene; the runtime path is correct.

### IN-03: Test mock `get` signature drops the `path` parameter, weakening Test 9's exact-match assertion as a structural guard

**File:** `test/tools/timer-status.test.ts:49, 68, 87, 124, 142, 228`

**Issue:** Most test mocks declare `get` as `async <T>(): Promise<T>` — no
`path` parameter — while the real signature is `get<T>(path: string):
Promise<T>`. TypeScript's structural compatibility allows the shorter form
to satisfy the type, but it means only Test 9 actually observes the path
the handler builds; every other test silently accepts whatever path the
handler computes.

That is fine in isolation, but if a future refactor (e.g., switching to a
`/{orgId}/time-entries?last=true` query path under D-2.5-05 reconsideration)
broke the path string while keeping the return shape correct, Tests 1-6 and
10 would all stay green. Only Test 9 would catch it.

**Fix:** Either (a) accept this as planner's discretion — Test 9 is the
named structural guard, the rest are payload tests — and add a one-line
comment in `buildClient` saying so, or (b) standardise on the full mock
signature `async <T>(_path: string): Promise<T>` so every test passively
asserts the handler at least *passes a string* into `get`.

Low priority; current setup matches the patterns in
`test/tools/entries-list.test.ts`.
