### WR-01 — Closed

The guard in `extractTimeEntry` now reads (line 61):

```ts
if (candidate === null || typeof candidate !== "object" || Array.isArray(candidate)) {
  return null;
}
```

The `Array.isArray(candidate)` clause closes the silent-acceptance hole for
`{ time_entry: [] }` and `{ time_entry: [{...}] }` drift shapes. Two new
regression tests cover both surfaces:

- **Test 11** (`timer-status.test.ts:241-262`) — empty-array drift collapses
  to `{ time_entry: null, is_running: false }`.
- **Test 12** (`timer-status.test.ts:264-286`) — non-empty-array drift with
  a wrapped `ongoing: true` entry MUST still produce `is_running: false`,
  pinning both dimensions (the array MUST be discarded before
  `entry?.ongoing` is read).

The fix is precisely scoped — `Array.isArray` is only applied to the
`time_entry` wrapper *value*, not to `raw` itself or to any field downstream
of the guard. Arrays passed at the top level (`raw = [{...}]`) are still
correctly rejected because the property lookup yields `undefined`, which
fails the `typeof candidate !== "object"` guard.

### IN-02: Test 4 hard-codes a sanitised body that masks the constructor contract

**File:** `test/tools/timer-status.test.ts:104-106`

**Status:** Open. Out of Plan 02.5-02 scope; planner's discretion.

**Issue:** The 404 test passes a literal body to the `KeepingApiError`
constructor:

```ts
throw new KeepingApiError(404, '{"error":{"message":"No entry found"}}');
```

The constructor contract says callers must pass the raw body through
`sanitiseBody(text, token)` first. The test happens to pass an already
token-free string, so the sanitisation invariant is not exercised. A future
maintainer copy-pasting this test for a new error case might inadvertently
pass an unsanitised body.

**Fix:** Either add a brief comment at the throw site noting the body must
already be sanitised, or call `sanitiseBody(rawBody, "fake-token")` inline
so the test demonstrates the contract. Low priority — pure documentation
hygiene; the runtime path is correct.

---

### IN-03: Test mock `get` signature drops the `path` parameter, weakening Test 9's exact-match assertion as a structural guard

**File:** `test/tools/timer-status.test.ts:49, 68, 87, 124, 142, 228` (and
now also 247, 270 from the new Tests 11 + 12, which follow the same
parameter-less shorthand)

**Status:** Open. Matches the established pattern in
`test/tools/entries-list.test.ts`; planner's discretion.

**Issue:** Most test mocks declare `get` as `async <T>(): Promise<T>` — no
`path` parameter — while the real signature is `get<T>(path: string):
Promise<T>`. TypeScript's structural compatibility allows the shorter form
to satisfy the type, but only Test 9 actually observes the path the
handler builds; every other test silently accepts whatever path the
handler computes. The two new tests (11 + 12) inherit this same shape.

**Fix:** Either accept this as planner's discretion — Test 9 is the named
structural guard, the rest are payload tests — and add a one-line comment
in `buildClient` saying so, or standardise on the full mock signature
`async <T>(_path: string): Promise<T>`. Low priority; current setup
matches the patterns in `test/tools/entries-list.test.ts`.
