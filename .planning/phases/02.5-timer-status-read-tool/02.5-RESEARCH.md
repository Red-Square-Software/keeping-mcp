## User Constraints (from CONTEXT.md)

### Locked Decisions

Verbatim from `.planning/phases/02.5-timer-status-read-tool/02.5-CONTEXT.md` §Implementation Decisions:

- **D-2.5-01:** Wire shape is `{ time_entry: <raw API entry>, is_running: boolean }`. The API's `time_entry` wrapper key is preserved literally (D-34-R pass-through spirit). `is_running` is added as a TOP-LEVEL sibling — not merged into the entry. SC #1 reads "raw `time_entry` object plus top-level `is_running`" — this shape is the literal match.
- **D-2.5-02:** `is_running` derivation is `time_entry?.ongoing === true`. Per SC #4: `ongoing: true` → `is_running: true`; `ongoing: false` → `is_running: false`; missing `ongoing` (or missing `time_entry`) → `is_running: false`. Defensive default — never throws on missing field.
- **D-2.5-03:** A 404 from the endpoint (documented in OpenAPI — "no matching entry") collapses to a graceful empty response: `{ time_entry: null, is_running: false }`. New users with zero historical entries get a usable answer rather than `isError`. Mirrors Phase 2's "graceful empty for feature-disabled orgs" pattern (META-01, META-02).
- **D-2.5-04:** A 200 OK whose body fails to yield a usable `time_entry` (key missing, `null`, or non-object) collapses to the same graceful empty shape as D-2.5-03. One empty-state surface, regardless of cause.
- **D-2.5-04a:** 404 detection happens by inspecting the thrown `KeepingApiError.status === 404` inside the tool handler's `catch`. All other errors (401, 403, 5xx, network) keep flowing through `toIsErrorContent()` unchanged. The empty-state branch is the ONLY special-case catch in the tool.
- **D-2.5-05a:** Strict wrapper read — the tool reads `raw.time_entry` verbatim and nothing else. No defensive multi-key fallback (no `entries[0]`, no bare-entry-as-object acceptance). entries-list.ts needed a tolerant normaliser because its schema was unverified pre-D-34-R; here the OpenAPI spec is authoritative and the wrapper is locked. Drift will be caught by the schema-drift test rather than masked by tolerant parsing.
- **D-2.5-05:** Pure-status semantics — the tool sends NO query parameters to `/time-entries/last`. Endpoint hit is bare `/{orgId}/time-entries/last`. Returns the absolute most-recent entry (running or stopped) for the authenticated user. The `is_running` boolean is what answers "is a timer running NOW?" — derived client-side from `ongoing`. Rejected alternative `?ongoing=true` because it would lose the most-recent-stopped-entry data Phase 3's `keeping_resume_timer` will need.
- **D-2.5-06:** Tool input schema is `{ organisation_id?: string }` — and nothing else. No `user_id`. No `date`. No `purpose`/`project_ids`/etc. The endpoint's other 8 documented filters are intentionally NOT exposed.
- **D-2.5-07:** File path `src/tools/timer-status.ts`, registration function `registerTimerStatus(server, client)`, registered by name `keeping_timer_status`. Pattern is byte-for-byte parallel to `src/tools/entries-list.ts`.
- **D-2.5-08:** Tool annotations are `readOnlyHint: true, destructiveHint: false, idempotentHint: true, openWorldHint: true` (matching every Phase 2 read tool; the openWorldHint is required because we call an external API).
- **D-2.5-09:** Dispatch via `client.get<unknown>(path)`. No new `KeepingClient` method, no new request strategy. `resolveOrgId(input.organisation_id)` resolves multi-org per D-26..29.
- **D-2.5-10:** Multi-org error envelope (D-27) and mid-session 401 envelope (D-25) are inherited unchanged. `MultiOrgError` and `KeepingApiError` surface through `toIsErrorContent()` as today.
- **D-2.5-11:** Unit tests are the only verification surface. No live-API probe needed (the endpoint is already proven by the Phase 2 probe-live run; the OpenAPI mirror at `.planning/research/keeping-openapi.json` documents the contract).
- **D-2.5-12:** Test fixture reuses the existing `test/fixtures/time-entry-response.sample.json` SHAPE as the structural reference. Tests construct mock `client.get` responses by wrapping a single fixture entry in `{ time_entry: ... }`. SC #5 satisfied.
- **D-2.5-13:** Ten test cases enumerated (see Phase Requirements section below).

## Summary

Phase 2.5 is the smallest possible vertical slice in the project: **one new tool file, one new test file, one new line in `src/server.ts`**. Every architectural decision is already locked in CONTEXT.md (D-2.5-01..13) and inherited from Phase 2's carry-forward (D-22..D-29, D-34-R). The Keeping API contract is locked via `.planning/research/keeping-openapi.json` (`GET /{organisation_id}/time-entries/last` returns `{ time_entry: <entry> }` on 200, `{ error: { message } }` on 403/404).

The single non-trivial choice is the empty-state collapse: a 404 (no entry exists) AND a 200-with-no-`time_entry` (defensive) both produce `{ time_entry: null, is_running: false }`. This is implemented as ONE additional catch branch in the tool handler (`err instanceof KeepingApiError && err.status === 404`) plus a strict-read normaliser (`raw.time_entry` only — no multi-key fallback per D-2.5-05a). The contrast with `entries-list.ts`'s tolerant normaliser is intentional and load-bearing: the OpenAPI spec authoritatively locks the wrapper shape here, so drift should fail loudly (caught by the schema-drift test pattern from D-36 / Pitfall 12), not be masked.

The tool consumes ONE new endpoint that Phase 2's `KeepingClient` already routes correctly via the D-34-R path strategy (`/{orgId}/...` — no `/organisations/` prefix). No new client method, no new request shape, no new error class. Phase 2.5 is approximately 35 LOC of source + ~250 LOC of tests + 1 import line + 1 register call.

**Primary recommendation:** Plan a single wave with a TDD pair: (W0) write `test/tools/timer-status.test.ts` covering the 10 D-2.5-13 cases against a `Partial<KeepingClient>` mock — copy `test/tools/entries-list.test.ts` skeleton; (W1) implement `src/tools/timer-status.ts` mirroring `src/tools/entries-list.ts` minus the URLSearchParams branching and minus the tolerant normaliser, plus the 404→empty catch arm; (W2) add one import + one `registerTimerStatus(...)` line to `src/server.ts`. Optional fourth slot for an `outputSchema` belongs in Claude's Discretion and should be deferred unless trivial.

**What:** The new tool is a near-verbatim sibling of `src/tools/entries-list.ts`, simplified by removing the URL branching and the tolerant normaliser, then augmented with the 404 catch branch.

**When to use:** All Phase 2.5 work follows this pattern. Sibling-by-substitution is the project's established read-tool convention (locked for Phase 2 — see STATE.md "Sibling-pattern copy locked (Plan 02-03)").

**Example (illustrative — planner refines):**

```typescript
// Source: src/tools/entries-list.ts (sibling) + locked decisions D-2.5-01..09
// This sketch is for planner reference; planner produces final PLAN.md tasks.

import type { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { z } from "zod";
import type { KeepingClient } from "../keeping/client.js";
import { KeepingApiError, toIsErrorContent } from "../keeping/errors.js";

const TimerStatusInput = z.object({
  organisation_id: z
    .string()
    .optional()
    .describe("Override KEEPING_ORG_ID; required for multi-org tokens."),
});

/**
 * Strict wrapper read per D-2.5-05a. Returns the raw inner entry only when
 * `raw.time_entry` is a non-null object. Anything else (missing key, null,
 * non-object) collapses to null — handled by the caller as graceful empty
 * per D-2.5-04.
 */
function extractTimeEntry(raw: unknown): Record<string, unknown> | null {
  if (raw === null || typeof raw !== "object") return null;
  const candidate = (raw as Record<string, unknown>).time_entry;
  if (candidate === null || typeof candidate !== "object") return null;
  return candidate as Record<string, unknown>;
}

export function registerTimerStatus(server: McpServer, client: KeepingClient): void {
  server.registerTool(
    "keeping_timer_status",
    {
      title: "Timer status (read-only)",
      description:
        "Returns the most recent time entry for the authenticated user plus a " +
        "derived `is_running` boolean indicating whether a Keeping timer is " +
        "currently running. Read-only; no API mutation. Use this to decide " +
        "whether a `keeping_stop_timer` or `keeping_resume_timer` call is " +
        "appropriate (Phase 3).",
      inputSchema: TimerStatusInput,
      annotations: {
        readOnlyHint: true,
        destructiveHint: false,
        idempotentHint: true,
        openWorldHint: true,
      },
      // outputSchema OMITTED per Claude's Discretion default (defer).
    },
    async (input) => {
      try {
        const orgId = await client.resolveOrgId(input.organisation_id);
        const raw = await client.get<unknown>(`/${orgId}/time-entries/last`);
        const entry = extractTimeEntry(raw);
        const is_running = entry?.ongoing === true;
        const payload = { time_entry: entry, is_running };
        return { content: [{ type: "text", text: JSON.stringify(payload, null, 2) }] };
      } catch (err) {
        // D-2.5-03 + D-2.5-04a: 404 = "no entry yet" = graceful empty.
        if (err instanceof KeepingApiError && err.status === 404) {
          const payload = { time_entry: null, is_running: false };
          return { content: [{ type: "text", text: JSON.stringify(payload, null, 2) }] };
        }
        // SAFE-04 + D-2.5-04a: every other error path (401/403/5xx/network/MultiOrg) → isError envelope.
        return toIsErrorContent(err);
      }
    },
  );
}
```

### Anti-Patterns to Avoid

- **Tolerant wrapper normaliser** (`time_entry || entries?.[0] || raw`) — explicitly rejected by D-2.5-05a. Use strict `raw.time_entry` read only. Drift is caught by tests, not masked by code.
- **Adding `KeepingClient.timerStatus()` or any new client method** — D-2.5-09 forbids new client surface. Dispatch via existing `client.get<unknown>`.
- **Adding a new error class** (e.g., `KeepingNotFoundError`) — D-2.5-04a explicitly assigns 404 detection to status-inspection on the existing `KeepingApiError`. No new error class.
- **Exposing OpenAPI filter params** (`user_id`, `date`, `purpose`, `project_ids`, etc.) — D-2.5-06 locks input to `{ organisation_id?: string }`. Deferred ideas are out of scope.
- **Forwarding 404 to `toIsErrorContent`** — D-2.5-03 says 404 is graceful empty, not an error. The single tool-handler catch branch is the ONLY special case.
- **Modifying `src/keeping/client.ts`, `src/keeping/types.ts`, or `src/keeping/errors.ts`** — CONTEXT.md §Files Phase 2.5 MUST NOT Touch is explicit.
- **Calling `console.log` anywhere in the tool body** — corrupts stdio framing (Pitfall 1). Biome `noConsole` enforces. The tool needs no logger calls.
- **Auto-retry on 404** — already handled correctly by `KeepingClient` (which only retries 429s for GETs). The 404 surfaces as `KeepingApiError` to the tool catch branch on the first attempt.
- **Padding the tool description with "you may use this to start a timer"** — that's `keeping_start_timer`'s job in Phase 3. This tool is read-only.

## Common Pitfalls

### Pitfall 1: Tolerant normaliser instinct

**What goes wrong:** A developer copying `entries-list.ts` line-by-line will reproduce `normaliseEntries` — a tolerant multi-key fallback — and call it `normaliseTimerStatus`. This silently masks API drift: if Keeping renames `time_entry` to `entry` or returns a bare object, the tool keeps working with stale assumptions instead of failing loudly.
**Why it happens:** entries-list.ts predates D-34-R; its tolerance was a pre-OpenAPI hedge. The locked OpenAPI contract removes the need.
**How to avoid:** Implement `extractTimeEntry` as a STRICT single-key read (`raw.time_entry` only). D-2.5-05a is explicit. Two tests cover this: D-2.5-13 #5 (`{ time_entry: null }`) and #6 (key missing). Both must produce the SAME empty-state output as the 404 catch branch — but via the STRICT extractor, not via the catch.
**Warning signs:** A `normalise*` function with multiple `if (Array.isArray(obj.X)) return ...` branches. Test 5 or 6 passes "for the wrong reason" (e.g., catching an `if (obj.entries)` clause that shouldn't exist).

### Pitfall 2: 404 caught by `toIsErrorContent`

**What goes wrong:** The catch block is structured as `catch (err) { return toIsErrorContent(err); }` (copy-pasted from entries-list.ts) and the 404 branch is forgotten. Users with no historical entries get `isError: true` instead of graceful empty.
**Why it happens:** entries-list.ts has NO special case in its catch — every error flows through `toIsErrorContent`. Phase 2.5 deliberately diverges with ONE branch.
**How to avoid:** D-2.5-13 #4 (404 from client) is the gate. The test must produce `res.isError === undefined/false` and parse the text content as `{ time_entry: null, is_running: false }`. The catch must check `err instanceof KeepingApiError && err.status === 404` BEFORE delegating to `toIsErrorContent`.
**Warning signs:** Test 4 fails with `expect(res.isError).toBeFalsy()` because the error fell through to the generic envelope.

### Pitfall 3: `is_running` truthy-evaluation instead of strict `=== true`

**What goes wrong:** `const is_running = entry?.ongoing` (without `=== true`) returns the raw value. If `ongoing` is missing, JavaScript yields `undefined`, and the JSON payload becomes `{ ..., is_running: undefined }` — which `JSON.stringify` strips entirely, leaving NO `is_running` key in the output. Wire shape violated; SC #1 fails.
**Why it happens:** Loose `is_running = entry?.ongoing ?? false` works in TypeScript but defaults `null`/`undefined` to `false` only — if the API returns `0`, `""`, or a truthy non-boolean it would be wrong.
**How to avoid:** Use strict `entry?.ongoing === true` (locked in D-2.5-02). The optional-chain handles "no entry"; the `=== true` enforces boolean strictness. Tests 1, 2, 3 cover all three branches.
**Warning signs:** `is_running` key missing from the serialised JSON in test output; or `is_running: true` slipping through for non-boolean `ongoing` values like `"yes"`.

### Pitfall 4: Empty-state output shape divergence between the two paths

**What goes wrong:** The 404 catch branch returns `{ time_entry: null, is_running: false }`. The strict-read normaliser branch (Test 5: `{ time_entry: null }`; Test 6: missing key) ALSO returns `{ time_entry: null, is_running: false }` — but a careless implementation could emit `{ time_entry: undefined, is_running: false }` (which serialises to `{ is_running: false }`). Tests would fail subtly.
**Why it happens:** TS `undefined` vs JSON `null` are different on the wire. `JSON.stringify({ time_entry: undefined })` produces `{}` — the key drops.
**How to avoid:** Always assign `time_entry: null` explicitly when there is no entry. The `extractTimeEntry` helper returns `null` (not `undefined`) on miss. Tests 4, 5, 6 all assert `parsed.time_entry === null` AND `parsed.is_running === false`.
**Warning signs:** Tests assert `expect(parsed).toEqual({ time_entry: null, is_running: false })` and fail because the parsed object is `{ is_running: false }` (key missing).

### Pitfall 5: Query string accidentally added to the URL

**What goes wrong:** A developer adds a `URLSearchParams` block from entries-list.ts and emits `/${orgId}/time-entries/last?` (trailing `?`) or `/${orgId}/time-entries/last?ongoing=true`. The endpoint still works (Keeping ignores empty/extra params) but D-2.5-05 is violated — and worse, `?ongoing=true` would filter to running timers ONLY, losing the most-recent-stopped-entry needed by Phase 3's `keeping_resume_timer`.
**How to avoid:** D-2.5-13 #9 asserts the path is exactly `/${orgId}/time-entries/last` — no `?`, no `=`. Verify with `expect(calls[0]).toBe("/47666/time-entries/last")` (or `toMatch(/^\/47666\/time-entries\/last$/)`).
**Warning signs:** Test 9 fails with the path containing `?` or `&`. Phase 3's `keeping_resume_timer` development surfaces unexpected behaviour because stopped entries are filtered out.

### Pitfall 6: Schema drift (Pitfall 12 carry-forward)

**What goes wrong:** Keeping renames `time_entry` to `entry` or returns the entry bare (no wrapper) in a future API version. Phase 2.5's strict reader returns `null` for every call. Users see "no timer" forever even when they have running timers.
**Why it happens:** The OpenAPI mirror at `.planning/research/keeping-openapi.json` is a point-in-time snapshot. The API can drift without notice.
**How to avoid:** D-2.5-05a explicitly accepts this risk — drift should fail loudly via the schema-drift test from D-36 (Pitfall 12) rather than be masked. The drift surface is: a non-empty fixture parsed against the actual response shape on every `npm run probe-live` re-run; the fixture-shape test catches divergence.
**Warning signs:** A future probe-live run produces a fixture where `time_entry` is renamed; Phase 2.5 tests all start failing with "key missing". This is the desired outcome.

### Pitfall 7: Annotation defaults inherited silently

**What goes wrong:** A developer omits the `annotations` block expecting reasonable defaults. The MCP spec's WRITE-tool defaults (`destructiveHint: true, idempotentHint: false`) apply, causing MCP clients to gate this READ tool behind confirmation dialogs.
**How to avoid:** Always set all four hints explicitly (D-2.5-08): `readOnlyHint: true, destructiveHint: false, idempotentHint: true, openWorldHint: true`. D-2.5-13 #10 asserts via `client.listTools()` reflection.
**Warning signs:** Test 10 fails with `expect(tool?.annotations?.readOnlyHint).toBe(true)` because the annotation is undefined.

### Pitfall 8: Empty-state response wrapped as `isError`

**What goes wrong:** A developer hand-rolls the empty-state branch as `return { isError: true, content: [...] }` to make the "no timer running" state easy to detect in the client. This violates D-2.5-03's "not an error" intent.
**How to avoid:** Graceful empty is NOT an error. Return `{ content: [{ type: "text", text: JSON.stringify({ time_entry: null, is_running: false }) }] }` — no `isError` key. Tests 4, 5, 6 assert `expect(res.isError).toBeFalsy()`.
**Warning signs:** Tests pass on text content but fail on `expect(res.isError).toBeFalsy()` — or worse, MCP client UX treats "no timer" as a tool failure.

### Phase-2.5-Specific Security Considerations

- **The 404-empty-state branch is the only special-case error handling.** A misimplementation here is a UX bug, not a security flaw — the tool still returns sanitised content via `toIsErrorContent` if the catch fails to match. There is no path where 404 reveals more than "no entry" via the `KeepingApiError.message` (which contains the localised `error.message` string from the Keeping API; safe to surface).
- **`KeepingApiError.message` for non-404 cases includes the sanitised body.** `sanitiseBody(text, token)` strips the bearer token already. The bodies are localised Keeping error strings (Dutch in tests), which are non-sensitive (no PII, no token).
- **The handler is `async` and awaits both `resolveOrgId` and `client.get`.** Both can throw; both throws are caught by the single `try/catch`. No unhandled-rejection surface.
