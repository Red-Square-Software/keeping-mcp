## Phase Goal

**As a** developer using a MCP client against keeping-mcp,
**I want to** call a single `keeping_timer_status` tool and receive the most-recent Keeping time entry plus a derived `is_running` boolean,
**so that** I can decide whether to log new hours, stop a running timer, or resume a stopped one — all without mutating any data.

<objective>
Ship the single vertical slice that satisfies Phase 2.5: a read-only MCP tool
`keeping_timer_status` backed by `GET /{org_id}/time-entries/last` plus a
client-side `is_running` boolean derived from the API's `ongoing` flag.

Purpose: Close the TIMER-01 status-read portion (start/stop/resume writes
remain in Phase 3 per D-33-R). The tool gives an AI client one read call to
answer "is a Keeping timer currently running, and if so on what?" — which is
the input Phase 3's `keeping_stop_timer` / `keeping_resume_timer` tools will
consume.
