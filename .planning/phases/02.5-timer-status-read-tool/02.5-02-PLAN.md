## Phase Goal (carried forward from 02.5-01-PLAN.md)

**As a** developer using a MCP client against keeping-mcp,
**I want to** call a single `keeping_timer_status` tool and receive the most-recent Keeping time entry plus a derived `is_running` boolean,
**so that** I can decide whether to log new hours, stop a running timer, or resume a stopped one — all without mutating any data.

<objective>
Close the single gap raised in `02.5-VERIFICATION.md`: `extractTimeEntry`
silently accepts array shapes as a valid `time_entry`, contradicting the
documented "bare-array collapses to null" contract (REVIEW.md WR-01) and
violating the locked D-2.5-05a strict-wrapper-read decision.

The fix is one line of production code (`Array.isArray(candidate)` added to
the existing guard) preceded by two RED unit tests that prove the current
implementation leaks arrays through to `payload.time_entry`.

Purpose: Honour the load-bearing rationale of D-2.5-05a — "Drift will be
caught by the schema-drift test rather than masked by tolerant parsing." Right
now `{ time_entry: [] }` is masked as `{ time_entry: [], is_running: false }`;
after this plan it correctly collapses to `{ time_entry: null, is_running: false }`
like every other non-object drift shape.

```typescript
function extractTimeEntry(raw: unknown): Record<string, unknown> | null {
  if (raw === null || typeof raw !== "object") return null;
  const candidate = (raw as Record<string, unknown>).time_entry;
  if (
    candidate === null ||
    typeof candidate !== "object" ||
    Array.isArray(candidate)
  ) {
    return null;
  }
  return candidate as Record<string, unknown>;
}
```

The change is purely narrowing: a value that was previously accepted

that was previously rejected becomes accepted. The 404 branch, the missing-
