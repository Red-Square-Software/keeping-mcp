## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User's `KEEPING_TOKEN` → `.env` file on disk | Token written to disk for the duration of the probe; never committed |
| Real API response → `.live-capture-raw.json` on disk | Raw data including project/client names exists locally; never committed |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06-01 | Information Disclosure | Token written to `.env` file lingers after probe | accept (user-visible) | `.env` is in `.gitignore` from Phase 1. User can delete the file after the probe if desired. The token is theirs; they generated it; revoke if compromised. |
| T-02-06-02 | Information Disclosure | Anonymise misses a Keeping-introduced field (e.g., `client_phone`) | mitigate | Task 2 is the human-review gate. If a field leaks, the user patches manually + reports the field name so the planner can extend `ANONYMISE_KEYS` in a follow-up plan. The denylist is intentionally extensible per CONTEXT discretion. |
| T-02-06-03 | Information Disclosure | LIVE-API.md narrative section accidentally quotes a description string | mitigate | Task 2 reviews this. The probe-live script's `buildLiveApiNotes()` was implemented to only list field NAMES + types — but a future patch could regress. Manual review catches it; if found, edit before commit. |

(2) Task 3 step 3 explicitly verifies `git status` does not show it. (3) `git add` of specific files in step 4 (not `git add -A`) prevents accidental inclusion even if gitignore were broken. |
