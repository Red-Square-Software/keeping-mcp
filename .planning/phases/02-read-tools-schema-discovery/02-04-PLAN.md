    ```yaml
    - name: Smoke test — MCP initialize handshake produces only valid JSON-RPC on stdout
      shell: bash
      env:
        KEEPING_TOKEN: kp_test_FAKE_token_value
      run: |
        set +e
        REQ='{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-11-25","capabilities":{},"clientInfo":{"name":"ci-smoke","version":"1.0.0"}}}'
        # Pipe initialize, capture stdout into OUTPUT and stderr to /tmp/init_stderr.
        # The server stays alive after initialize; on POSIX the closed-stdin EOF makes the
        # transport exit. On Windows Git Bash this behaviour is the same per Phase 1 D-14.
        OUTPUT=$(printf '%s\n' "$REQ" | node dist/bin/keeping-mcp.js 2>/tmp/init_stderr)
        EXIT_CODE=$?
        STDERR=$(cat /tmp/init_stderr)
        set -e

        # Assertion 1: stdout MUST be one or more valid JSON-RPC lines (parse each).
        if [ -z "$OUTPUT" ]; then
          echo "FAIL: no stdout produced; server did not respond to initialize"
          echo "stderr was: $STDERR"
          exit 1
        fi
        while IFS= read -r LINE; do
          [ -z "$LINE" ] && continue
          echo "$LINE" | node -e "JSON.parse(require('fs').readFileSync(0,'utf8'))" >/dev/null 2>&1 || {
            echo "FAIL: stdout contained non-JSON line: $LINE"
            exit 1
          }
        done <<< "$OUTPUT"

        # Assertion 2: stderr MUST NOT contain the fake token (D-08 redaction + AUTH-03).
        if echo "$STDERR" | grep -qF "kp_test_FAKE_token_value"; then
          echo "FAIL: fake KEEPING_TOKEN leaked to stderr"
          echo "stderr was: $STDERR"
          exit 1
        fi

        # Assertion 3: First stdout line MUST be a valid initialize response.
        FIRST=$(printf '%s' "$OUTPUT" | head -n1)
        echo "$FIRST" | node -e "
          const r = JSON.parse(require('fs').readFileSync(0,'utf8'));
          if (r.jsonrpc !== '2.0' || r.id !== 1 || !r.result) {
            console.error('FAIL: not a valid JSON-RPC response to id=1:', JSON.stringify(r));
            process.exit(1);
          }
          if (!r.result.serverInfo || r.result.serverInfo.name !== 'keeping-mcp') {
            console.error('FAIL: serverInfo.name is not keeping-mcp:', JSON.stringify(r));
            process.exit(1);
          }
          if (!r.result.protocolVersion) {
            console.error('FAIL: missing protocolVersion in initialize response:', JSON.stringify(r));
            process.exit(1);
          }
        " || exit 1

        echo "Initialize smoke PASSED: clean stdout, no token leak, valid initialize response"
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tool input `from` / `to` strings → URL query parameter | User-controlled date strings reach the API URL after Zod regex validation |
| Built binary stdout → CI assertion | Any stray output from new deps or accidental `console.log` would corrupt the JSON-RPC stream |
| Env `KEEPING_TOKEN` → process → fetch Authorization header / logger redaction | Token must never appear on stderr even when injected at CI time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04-01 | Tampering | `from` / `to` / `user_id` strings flowing into `URLSearchParams` | mitigate | Zod regex `^\d{4}-\d{2}-\d{2}$` on date fields rejects anything that isn't a calendar date BEFORE URL construction. `user_id` is set via `params.set()` (key/value safe API). `URLSearchParams` handles encoding. |
| T-02-04-02 | Information Disclosure | A dependency added in Plan 02-01 prints something to stdout at import time (Pitfall F) | mitigate | Task 2's smoke step (Assertion 1) catches non-JSON on stdout immediately. CI matrix [ubuntu, windows] × [22, 24] amplifies the catch. |
| T-02-04-03 | Information Disclosure | A future code change logs `process.env.KEEPING_TOKEN` accidentally | mitigate | Task 2's smoke step (Assertion 2) greps stderr for the fake token and fails if found. AUTH-03 regression gate. |
| T-02-04-04 | Denial of Service | `keeping_list_entries` called with a year-long date range returning 5-10 MB of JSON exceeds MCP message-size limits (Pitfall E) | mitigate | Default `limit: 200`; hard cap `max: 1000` in Zod. Description tells callers to narrow date range for larger windows. Phase 2 does not implement pagination — Open Question #2 will be revisited in Phase 3 after probe-live captures pagination shape. |
| T-02-04-05 | Spoofing | A malicious MCP client supplies `user_id` to read another user's entries | accept (server-side scope) | Token is scoped to the authenticated user; Keeping's API enforces access control server-side. If a non-admin token requests another user's entries, the API returns 4xx — surfaces via `toIsErrorContent`. v1 explicitly does not support admin-on-behalf-of writes (REQUIREMENTS.md Out of Scope). |

</threat_model>
