<objective>
Wave 2 of Phase 2: bring the MCP server to life. This is the **first vertical slice** under
MVP mode — after this plan completes, a real user can invoke `keeping_me` from any MCP client
against their real Keeping account and get back their authenticated identity. The slice runs
top-to-bottom: `bin/keeping-mcp.ts` boots `StdioServerTransport` → `src/server.ts` registers
the first tool → `src/tools/me.ts` calls `client.me()` + `client.resolveOrgId()` → `KeepingClient`
makes the real HTTPS call through `p-throttle` + `p-retry` → returns the authenticated user.

The slice intentionally includes the full `KeepingClient` (throttle, retry, cache, sanitisation,
resolveOrgId) — not because we need all of it for `keeping_me` alone, but because every
remaining tool in Plans 02-03 and 02-04 calls this same client and would otherwise need to
revisit its shape. Land the contract once.

Purpose: prove the boot path works, prove identity caching works, prove the throttle/retry
composition works, prove the 401 path emits the D-25 wording — all in one slice that delivers a
runnable, user-callable tool.
Output: extended entrypoint, new `src/server.ts`, full `KeepingClient`, `keeping_me` tool, and
two test files (KeepingClient unit + `keeping_me` InMemoryTransport).

KeepingClient `rawFetch` skeleton (RESEARCH §Pattern 2 lines 413-439):
- BASE = "https://api.keeping.nl/v1"
- TIMEOUT_MS = 10_000
- AbortSignal.timeout(TIMEOUT_MS)
- Authorization: Bearer ${token}, Content-Type/Accept: application/json
- 401 → throw new KeepingAuthError()
- 429 → throw new KeepingRateLimitError(Number(headers.get('Retry-After') ?? '30'))
- !ok → throw new KeepingApiError(res.status, sanitiseBody(await res.text(), token))
- ok → res.json()

KeepingClient `request<T>` composition (RESEARCH §Pattern 2 lines 442-462):
- throttle wraps rawFetch; pRetry wraps the throttled fn
- retries: 3
- shouldRetry: method === 'GET' && (error instanceof KeepingRateLimitError || error instanceof TypeError)
- onFailedAttempt: async — if KeepingRateLimitError, sleep retryAfter*1000 ms then let p-retry continue

resolveOrgId (RESEARCH §Pattern 3 lines 501-514):
- await organisations() to get cached list
- candidate = input ?? process.env.KEEPING_ORG_ID
- if candidate: must be in ids[] else throw MultiOrgError(orgs); D-29 catches typos
- else if ids.length === 1: return ids[0]
- else: throw MultiOrgError(orgs); D-27 wording lives on the error class itself

Tool registration shape (RESEARCH §Pattern 1 lines 339-371, Pitfall B):
- Always `inputSchema: z.object({...})` (never a raw shape)
- annotations: { readOnlyHint: true, destructiveHint: false, idempotentHint: true, openWorldHint: true }
- handler: try { ... return content } catch (err) { return toIsErrorContent(err) }
- NEVER throw out of a handler

McpServer construction (PATTERNS lines 97-113, Pitfall A):
- new McpServer({ name: "keeping-mcp", version: "0.1.0" })
- DO NOT declare `capabilities: { logging: ... }` — stderr remains the only logging surface (D-09)
- Add a comment at the construction site documenting why

bin/keeping-mcp.ts extension (PATTERNS lines 60-71):
- Keep existing loadConfig + createLogger lines verbatim
- Replace the `log.info(...) + process.exit(0)` tail with:
  - `const client = new KeepingClient(config.KEEPING_TOKEN, log);`
  - `const server = createServer(client, config, log);`
  - `await server.connect(new StdioServerTransport());`
- Top-level await is allowed (ESM + Node >=22)
- No process.exit at the end — the transport owns the event loop

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client (Claude Code) → server | JSON-RPC over stdio; `initialize` is the first message |
| Tool input arguments → `client.resolveOrgId(input)` | User-controlled string flows into URL path construction |
| `KeepingClient` → `api.keeping.nl` | HTTPS request crosses out of process; token in `Authorization` header |
| API response body → error message → tool isError text → MCP client | Path that could leak the token if any error class fails to scrub |
| `process.env.KEEPING_TOKEN` → KeepingClient field → fetch headers | Token never reaches any other surface |

| T-02-02-01 | Information Disclosure | `Authorization` header echoed in error response body, then captured into `KeepingApiError` | mitigate | `sanitiseBody(text, this.token)` runs before `new KeepingApiError(status, body)` in `rawFetch` (Pitfall G).
