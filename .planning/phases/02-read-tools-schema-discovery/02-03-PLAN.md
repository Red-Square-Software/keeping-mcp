<!-- Graceful-empty pattern for projects/tasks (META-01, META-02; RESEARCH lines 106-107) -->

When the Keeping API returns a 404 for `/organisations/:id/projects` OR `/organisations/:id/tasks`,
the org has the feature disabled. This is NOT a failure. Handler must:
1. Catch `KeepingApiError` and check `err.status === 404`.
2. Return `{ content: [{ type: "text", text: "Projects feature not enabled for this organisation." }] }`.
   - For `keeping_tasks`: text is `"Tasks feature not enabled for this organisation."`.
3. Do NOT set `isError: true` on this path тАФ feature-off is not an error.

If the API instead returns `{ projects: [] }` (empty success), surface as a normal empty array
inside the payload, NOT as a feature-disabled message. Distinguish by HTTP status, not body.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tool input `organisation_id` тЖТ `client.resolveOrgId` тЖТ URL path interpolation | User-controlled string flows into `/organisations/${id}/projects` URL |
| API 404 тЖТ tool handler тЖТ user-facing text | Distinguishing "feature disabled" from "real failure" by status code |
| KeepingClient error тЖТ tool isError text тЖТ MCP client | Same error sanitisation path as Plan 02-02; defence-in-depth preserved |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Tampering | `organisation_id` input is interpolated into URL path | mitigate | `client.resolveOrgId(input)` validates the id against the cached `organisations()` list (D-29). Any id not in the list throws `MultiOrgError` BEFORE the URL is constructed. Defence-in-depth: even if the validation regressed, `KeepingApiError` from the API would surface as a sanitised isError тАФ no shell injection risk because it's a URL path, not a shell command. |
| T-02-03-02 | Information Disclosure | Real API failure surfaces error body via `toIsErrorContent` | mitigate | KeepingApiError body was scrubbed via `sanitiseBody(text, token)` in `KeepingClient.rawFetch` (Plan 02-02, Pitfall G). Tool handler only echoes `err.message` which is post-scrub. |
| T-02-03-03 | Spoofing | A 404 on `/organisations/:id/projects` could ALSO mean the org id is wrong, not just feature-disabled | accept | The graceful-empty message is informational and harmless even if the org is wrong тАФ `resolveOrgId` would have caught a bad id already (D-29). A 404 here genuinely indicates feature-off. If Keeping changes its behaviour, the message remains accurate ("not enabled" covers both "feature off" and "endpoint absent"). |
| T-02-03-04 | Information Disclosure | Project / task names may include internal project codes тАФ these reach the MCP client verbatim | accept | The MCP client is local to the user (Claude Code on the same machine). The user is authenticated to view their own project list; there is no broader audience. Raw pass-through is intentional per the META-01/02 spec. |

</threat_model>
