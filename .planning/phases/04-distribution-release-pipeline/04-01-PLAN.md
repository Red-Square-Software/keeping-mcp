<objective>
Create `server.json` (the MCP Registry manifest) and `scripts/check-publish-shape.ts` (the pre-publish gate that protects DIST-04's "no leak via files whitelist" guarantee and locks the mcpName↔server.json.name binding for DIST-05). Wire the gate into a `check-publish-shape` npm script so Plan 04-03's release workflow can call it without bash heredoc gymnastics.

Purpose: Phase 4 publishes to two registries (npm + MCP Registry). Both reject any drift between `package.json.mcpName` and `server.json.name`. The tarball allowlist (DIST-04 / ROADMAP SC #1) is currently held by the existing `files[]` array but has no automated guard — a careless future addition (e.g., `"scripts"` or `"test"` to `files[]`) would silently ship secrets. This plan ships the file shape AND the assertion that guards it.

Output: `server.json` (committed), `scripts/check-publish-shape.ts` (Node-runnable via tsx), `package.json.scripts.check-publish-shape` entry. Zero runtime changes to the MCP server itself.
</objective>

<research_excerpts>
From 04-RESEARCH.md §Pattern 2 (server.json minimum viable for stdio + npm):
- `$schema`: "https://static.modelcontextprotocol.io/schemas/2025-12-11/server.schema.json"
- `name`: "io.github.red-square-software/keeping-mcp"
- `description`: 1–100 chars
- `version`: placeholder "0.0.0" (CI rewrites at publish time)
- `repository.url`: "https://github.com/red-square-software/keeping-mcp" + `repository.source`: "github"
- `websiteUrl`: same as repository
- `packages[0]`: { registryType: "npm", identifier: "keeping-mcp", version: "0.0.0", transport: { type: "stdio" } }

From 04-RESEARCH.md §Pitfall 6 (`mcpName` ↔ server.json.name desync):
> `mcp-publisher publish` validates that the npm package's `mcpName` field matches the server.json's `name` field. If they don't, publish fails with a namespace verification error.
Mitigation: equality assert in CI before publish.

From 04-RESEARCH.md §Pattern 1, Step 3 (allowlist assertion):
> Use `npm pack --dry-run --json | jq -r '.[0].files[].path' | sort` and compare to a literal allowlist; fail if mismatched.
We adapt this to Node (`tsx`) for portability (no `jq` on Windows runners) and so the same script is callable from a developer's shell as `npm run check-publish-shape`.
</research_excerpts>

| repo → MCP Registry | server.json carries the namespace claim. Drift between `package.json.mcpName` and `server.json.name` breaks the namespace verification at `mcp-publisher publish` time. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-04-01-01 | Information Disclosure | `package.json.files[]` whitelist | mitigate | `scripts/check-publish-shape.ts` Assertion 1 — exact-set match against `ALLOWLIST` constant; any drift fails CI before `npm publish` runs (Plan 04-03 wires it as a release-workflow step). |
| T-04-01-02 | Spoofing | MCP Registry namespace `io.github.red-square-software/keeping-mcp` | mitigate | `scripts/check-publish-shape.ts` Assertion 2 — `package.json.mcpName === server.json.name` equality. Namespace verification in `mcp-publisher publish` would catch it too, but only after a tag push; local guard catches it pre-tag. |
| T-04-01-03 | Tampering | `.npmignore` silently overrides `files[]` | mitigate | `scripts/check-publish-shape.ts` Assertion 3 — refuses to run if `.npmignore` exists. Belt-and-suspenders for T-04-01-01. |
| T-04-01-SC | Tampering | npm package supply chain | accept | No new npm dependencies added by this plan (reuses existing `tsx`). RESEARCH §Package Legitimacy Audit: "Not applicable to Phase 4." |
