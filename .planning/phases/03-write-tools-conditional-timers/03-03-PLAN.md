<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client → registerUpdateEntry handler | Zod parses `entry_id` as `z.number().int().positive()` — path-traversal vector for `${entry_id}` template literal blocked at the schema layer. |
| Tool handler → KeepingClient.patch | Path `/${orgId}/time-entries/${entry_id}` — both segments are numeric strings from Zod or `resolveOrgId`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Tampering / Spoofing | `entry_id` path segment | mitigate | `entry_id: z.number().int().positive()` rejects strings, negatives, decimals. Test 1 covers the happy path; Zod's built-in rejection covers the invalid cases. |
| T-03-03-02 | Elevation of Privilege | `date`, `purpose`, `user_id` fields in update | mitigate | Zod schema simply does not declare these fields. Extra fields stripped by Zod. Even if a client sneaks them in, they never reach the body builder. Test 6 confirms a `date` field in input does NOT appear in posted body. |
| T-03-03-03 | Tampering | PATCH semantics — accidental nullification | mitigate | Body builder uses `if (input.X !== undefined) body.X = input.X` for every field — undefined fields are SKIPPED (not null'd, not zero'd). Test 4 confirms a single-field PATCH sends only that field. |
| T-03-03-04 | Tampering | Same AND-gate inversion as add-entry | mitigate | Same `previewOrCall` consumer; same `confirm === true` strict check. Test 1 + Test 3 cover both AND-gate cases. |
| T-03-03-05 | Information Disclosure | KeepingApiError + token | mitigate | Same Phase 2 carry-forward sanitisation as add-entry. |
| T-03-03-SC | Tampering | npm installs | accept | No installs. |
</threat_model>

    Handler body:
    ```
    async (input) => {
      try {
        const orgId = await client.resolveOrgId(input.organisation_id);
        const path = `/${orgId}/time-entries/${input.entry_id}`;

        // PATCH partial — body contains ONLY supplied fields. No defaults.
        const body: Record<string, unknown> = {};
        if (input.project_id !== undefined) body.project_id = input.project_id;
        if (input.task_id !== undefined) body.task_id = input.task_id;
        if (input.note !== undefined) body.note = input.note;
        if (input.tag_ids !== undefined) body.tag_ids = input.tag_ids;
        if (input.external_references !== undefined) body.external_references = input.external_references;
        if (input.start !== undefined) body.start = input.start;
        if (input.end !== undefined) body.end = input.end;
        if (input.hours !== undefined) body.hours = input.hours;

        const result = await previewOrCall<{ time_entry: unknown; meta?: unknown }>(
          client,
          { requireConfirm: config.KEEPING_REQUIRE_CONFIRM, confirm: input.confirm === true },
          { method: "PATCH", path, body },
        );
        return { content: [{ type: "text", text: JSON.stringify(result, null, 2) }] };
      } catch (err) {
        if (classifyAmbiguous(err)) {
          const msg = err instanceof Error ? err.message : String(err);
          return { isError: true, content: [{ type: "text", text: `${AMBIGUOUS_TEXT} (${msg})` }] };
        }
        return toIsErrorContent(err);
      }
    }
    ```

    - No defaulting. Empty body is technically allowed (Keeping returns the entry unchanged). Optional UX improvement (NOT REQUIRED for tests to pass): if `Object.keys(body).length === 0`, return an isError envelope explaining at least one field is required.
