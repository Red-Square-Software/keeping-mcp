<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client → registerDeleteEntry handler | Zod `entry_id: z.number().int().positive()` blocks path-traversal vectors. Same mitigation as 03-03. |
| Tool dry-run → client.get → response body → would_delete echo | The fetched entry is echoed verbatim into the tool response. Echoes `note` if present — by design (user reviews preview). Same trade-off documented in 03-02 threat model. |
| Tool confirm → client.delete → null/empty body | 204 tolerant per 03-01; tool wraps null as `{ ok: true }`. No PII surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04-01 | Tampering / Spoofing | `entry_id` path segment | mitigate | `z.number().int().positive()` Zod guard. Same as 03-03 T-03-03-01. |
| T-03-04-02 | Elevation of Privilege | Deleting other users' entries | mitigate | API enforces ownership: non-admin tokens cannot DELETE another user's entries (per OpenAPI + D-3-10 design). Tool surfaces 403 verbatim. No new code-side check needed. Documented in handler header comment for traceability. |
| T-03-04-03 | Repudiation | Confirm-path bypass via env-false | accept | Same env-false escape hatch as the other write tools. README warning is REL-05's job (Phase 4). |
| T-03-04-04 | Information Disclosure | Extra GET on dry-run as oracle | mitigate | The dry-run GET happens ONLY when `requireConfirm && !confirm`. The user requesting a dry-run is by construction authenticated and reviewing entries they own. The GET path is the same as Keeping's own UI would call. No new attack surface. |
| T-03-04-05 | Tampering | Confirm path accidentally calling GET first | mitigate | Tool code structure: an `if (isDryRun)` branch contains the GET; the confirm branch does NOT call get. Test 2 + Test 3 assert `gets.length === 0` on the confirm path. |
| T-03-04-SC | Tampering | npm installs | accept | No installs. |
</threat_model>

    ```
    async (input) => {
      try {
        const orgId = await client.resolveOrgId(input.organisation_id);
        const path = `/${orgId}/time-entries/${input.entry_id}`;
        const isDryRun = config.KEEPING_REQUIRE_CONFIRM && input.confirm !== true;

        if (isDryRun) {
          // Extra GET only on preview — never on the actual delete (D-3-03).
          const wouldDelete = await client.get<unknown>(path);
          return {
            content: [{
              type: "text",
              text: JSON.stringify({
                would_post: {
                  method: "DELETE",
                  url: `https://api.keeping.nl/v1${path}`,
                  body: null,
                },
                would_delete: wouldDelete,
              }, null, 2),
            }],
          };
        }

        // Confirm path: delegate through the gate (which will call client.delete since confirm===true).
        // The rawFetch 204 fix from 03-01 means client.delete resolves with null on success;
        // surface the wrapped result so the user sees "ok: true" rather than a bare null.
        const result = await previewOrCall<unknown>(
          client,
          { requireConfirm: config.KEEPING_REQUIRE_CONFIRM, confirm: input.confirm === true },
          { method: "DELETE", path },
        );
        return {
          content: [{ type: "text", text: JSON.stringify(result ?? { ok: true }, null, 2) }],
        };
      } catch (err) {
        if (classifyAmbiguous(err)) {
          const msg = err instanceof Error ? err.message : String(err);
          return { isError: true, content: [{ type: "text", text: `${AMBIGUOUS_TEXT} (${msg})` }] };
        }
        return toIsErrorContent(err);
      }
    }
    ```
