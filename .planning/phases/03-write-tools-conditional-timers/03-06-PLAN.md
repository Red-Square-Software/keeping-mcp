<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client → registerStopTimer handler | Zod `entry_id: z.number().int().positive()` blocks path-traversal. |
| KeepingClient.requestWithHeaders → fetch → response headers → tool output | `X-Server-Time-Ms` from response is parsed and surfaced. Untrusted-network value; tool MUST validate (Number.isFinite, > 0) before using as elapsed-time anchor. |
| Token in Authorization header (NOT response headers) | The Headers object the tool reads is `response.headers`, NOT `request.headers`. No token surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06-01 | Tampering | `X-Server-Time-Ms` header injection | mitigate | Tool parses via `const ms = Number(headers.get("X-Server-Time-Ms")); const valid = Number.isFinite(ms) && ms > 0;`. Invalid → fall back to `Date.now()` + `log.warn`. Test 4 covers `"abc"` → fallback path; Test 5 covers missing header → fallback path. |
| T-03-06-02 | Tampering | Missing X-Server-Time-Ms swallowed as `NaN` propagating into client output | mitigate | Same Number.isFinite gate. Without it, `Number("abc")` → `NaN`, would crash downstream consumers. Test 4 asserts `server_time_ms` is a positive finite number in the fallback path (not NaN, not null). |
| T-03-06-03 | Tampering | Verb mistake — POST instead of PATCH | mitigate | D-3-05 supersedes D-32-R. Test 2 strict-asserts the call passed `"PATCH"` to requestWithHeaders. |
| T-03-06-04 | Information Disclosure | `client.log.warn` accidentally including token | mitigate | The warn message is a fixed string with no body interpolation: `"X-Server-Time-Ms header missing on stop response; falling back to local clock"`. logger.ts already runs `.replaceAll(token, "***")` at emit time as defence-in-depth. |

    Handler body (inline-gate pattern):
    ```
    async (input) => {
      try {
        const orgId = await client.resolveOrgId(input.organisation_id);
        const path = `/${orgId}/time-entries/${input.entry_id}/stop`;
        const isDryRun = config.KEEPING_REQUIRE_CONFIRM && input.confirm !== true;

        if (isDryRun) {
          return {
            content: [{
              type: "text",
              text: JSON.stringify({
                would_post: {
                  method: "PATCH",
                  url: `https://api.keeping.nl/v1${path}`,
                  body: null,
                },
              }, null, 2),
            }],
          };
        }

        const { body, headers } = await client.requestWithHeaders<{ time_entry: unknown; meta?: unknown }>("PATCH", path);
        const headerValue = headers.get("X-Server-Time-Ms");
        const parsed = Number(headerValue);
        let server_time_ms: number;
        if (Number.isFinite(parsed) && parsed > 0) {
          server_time_ms = parsed;
        } else {
          client.log.warn("X-Server-Time-Ms header missing on stop response; falling back to local clock");
          server_time_ms = Date.now();
        }

        return {
          content: [{ type: "text", text: JSON.stringify({ ...body, server_time_ms }, null, 2) }],
        };
      } catch (err) {
        if (classifyAmbiguous(err)) {
          const msg = err instanceof Error ? err.message : String(err);
          return { isError: true, content: [{ type: "text", text: `${AMBIGUOUS_TEXT} (${msg})` }] };
        }
        return toIsErrorContent(err);
      }
    }
    ```

    Note: spreading `...body` into the response keeps the `time_entry` + `meta` wrapper visible AND adds `server_time_ms` as a sibling. This is the structure Test 2 asserts.
